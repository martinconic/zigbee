# zigbee 0.5.1 — `--bootnode` flag

Tiny patch on top of [`0.5.0`](0.5.0.md). Adds a `--bootnode` flag
that accepts the same multiaddr forms bee uses in
`testnet.yaml`, so you can point zigbee at a public bootnode
(by hostname) without first having to run `zigbee resolve` and
copy an IP into `--peer`.

## What's new

```
--bootnode MA       bootnode multiaddr; one of:
                      /dnsaddr/<host>           DNS-resolved
                      /ip4/<x>/tcp/<y>[/p2p/<id>]
                    For /dnsaddr/ zigbee resolves the TXT
                    records and tries each candidate in turn
                    until one connects (the others stay in
                    the table for hive). Mutually exclusive
                    with --peer.
```

Previously the only way to dial a peer was `--peer ip:port`,
which required you to do the DNS work yourself:

```bash
# 0.5.0 — two-step
zigbee resolve sepolia.testnet.ethswarm.org
# eyeball the output, copy an IP …
zigbee --peer 167.235.96.31:32491 --network-id 10 daemon ...
```

```bash
# 0.5.1 — one step
zigbee --bootnode /dnsaddr/sepolia.testnet.ethswarm.org \
       --network-id 10 daemon ...
```

## Why two flags instead of overloading `--peer`

`--peer` keeps its precise meaning: dial *this exact peer*, no
resolution. Useful for the local-bee dev loop
(`--peer 127.0.0.1:1634`) and for "force-connect to X."
`--bootnode` is the bootstrap-with-fallback flag — you give it
a hostname or multiaddr, it expands to a candidate list, and
the daemon tries each in order until one handshakes.

This mirrors bee's own config: `testnet.yaml` uses
`bootnode: [/dnsaddr/...]` for the DNS-resolved bootstrap entry
and has no `peer:` field at all. Anyone who's read bee's config
recognises `--bootnode` immediately.

## Implementation details

- New `Args.bootnode_arg: ?[]const u8` and `Args.peer_explicit: bool`
  in `src/main.zig`. `parseArgs` rejects passing both `--peer` and
  `--bootnode` (`error.ConflictingPeerFlags`).
- New `BootnodeCandidate { ip, port }` and `BootnodeCandidates`
  (arena-owned list) types in `src/main.zig`.
- New `resolveBootnodeCandidates(allocator, raw)` helper. Accepts:
  - `/dnsaddr/<host>` — calls `dnsaddr.resolve`, parses each
    returned multiaddr with `multiaddr.Multiaddr.fromText`, extracts
    `(ip4, tcp)` pairs.
  - `/ip4/<x>/tcp/<y>[/p2p/<id>]` — single literal multiaddr;
    extracts `(ip4, tcp)`, drops `/p2p/...`.
  - Anything else (bare `host:port`, `/dns4/...`) →
    `error.UnsupportedBootnodeForm`.
- `extractIp4Tcp(multiaddr)` walks an iterator looking for the
  `ip4` and `tcp` components, returning null on missing/malformed.
  `/p2p/`, `/udp/`, etc. are skipped without erroring.
- `P2PNode.DaemonOpts` now takes `bootnodes: []const BootnodeCandidate`
  instead of a single `(bootnode_ip, bootnode_port)` pair.
- `P2PNode.daemonRun` walks the candidate list in order; first
  successful `Connection.dial` becomes the bootstrap connection.
  On dial failure it logs the error and tries the next candidate.
  If every candidate fails: `error.AllBootnodesUnreachable`.

## Tests

6 new unit tests in `src/main.zig`:
- `parseArgs: --bootnode parses /dnsaddr/`
- `parseArgs: --peer + --bootnode is rejected`
- `resolveBootnodeCandidates: literal /ip4/.../tcp/...`
- `resolveBootnodeCandidates: /ip4/.../tcp/.../p2p/... drops the p2p suffix`
- `resolveBootnodeCandidates: rejects bare host:port`
- `resolveBootnodeCandidates: rejects /dnsaddr/ with empty host`

The `/dnsaddr/` resolution path itself is exercised by the
existing `dnsaddr.zig` tests; the new tests cover the
parse + dispatch logic without needing a live DNS round-trip.

Total suite: **113/113 tests pass** (was 107).

## Backwards compatibility

Fully backwards compatible. Every `--peer ip:port` invocation that
worked in 0.5.0 still works in 0.5.1 unchanged. The default
(no flag) is still `--peer 127.0.0.1:1634`.

## Not changed

- The `resolve` subcommand still exists for explicit lookups
  (`zigbee resolve sepolia.testnet.ethswarm.org` prints the
  multiaddrs and exits). Useful for inspection or for piping
  into shell scripts.
- The retrieve subcommand (one-shot) accepts `--bootnode` too,
  but only uses the *first* resolved candidate — single-shot
  retrieval doesn't try further candidates on dial failure.
  Daemon mode is the case that benefits from the candidate list.
