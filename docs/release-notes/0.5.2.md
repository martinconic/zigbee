# zigbee 0.5.2 — cross-compile build, first multi-platform binary release

Build-system patch on top of [`0.5.1`](0.5.1.md). No runtime
behaviour change for the existing `x86_64-linux` use case — but the
release pipeline now produces pre-built binaries for six platforms,
so users no longer need a Zig toolchain just to run zigbee.

## What's new for users

Pre-built binaries attached to the GitHub Release page:

| Filename | Target | Notes |
|---|---|---|
| `zigbee-linux-amd64-musl` | `x86_64-linux-musl` | Statically linked. Single-file drop on any Linux x86_64 (Alpine, Ubuntu, RHEL, …). |
| `zigbee-linux-arm64-musl` | `aarch64-linux-musl` | Statically linked. ARM64 servers, Raspberry Pi 4/5, Jetson, etc. |
| `zigbee-linux-arm64` | `aarch64-linux-gnu` | Dynamically linked (glibc). Slightly smaller; needs a glibc-based distro. |
| `zigbee-linux-armv7-musl` | `arm-linux-musleabihf` | Statically linked, 32-bit. Pi Zero W, older Pi models, embedded ARM Linux. |
| `zigbee-darwin-amd64` | `x86_64-macos` | Intel Mac. |
| `zigbee-darwin-arm64` | `aarch64-macos` | Apple Silicon (M-series). |
| `SHA256SUMS` | — | sha256 of every binary above. |

ReleaseSafe optimization mode (safety checks on, optimised). Sizes
range from 1.9 MB (macOS) to 6.8 MB (static Linux).

To use:

```bash
curl -L -o zigbee https://github.com/martinconic/zigbee/releases/download/v0.5.2/zigbee-linux-amd64-musl
chmod +x zigbee
./zigbee --bootnode /dnsaddr/sepolia.testnet.ethswarm.org --network-id 10 daemon
```

## What's new under the hood

### `build.zig` compiles libsecp256k1 from C sources

Before 0.5.2 the build linked a host-built static archive at
`vendor/secp256k1/build/lib/libsecp256k1.a`, produced by the
upstream autotools/CMake setup. That archive was machine-specific —
running `zig build -Dtarget=aarch64-linux-musl` would happily link an
x86_64 archive against ARM code and fail at the linker step (or
worse, succeed with garbage).

`build.zig` now compiles libsecp256k1 in-tree via `zig cc`. Three
C files do the whole job:

```
vendor/secp256k1/src/secp256k1.c
vendor/secp256k1/src/precomputed_ecmult.c
vendor/secp256k1/src/precomputed_ecmult_gen.c
```

with `-DENABLE_MODULE_RECOVERY=1`. zigbee uses recoverable ECDSA
only — the optional ECDH / schnorr / musig / ellswift / extrakeys
modules in upstream are not compiled in (small binary-size win).

This means **`zig build -Dtarget=<any>`** Just Works for any target
zig supports: cross-compilation needs nothing on the host machine
beyond zig itself.

### `src/store.zig` alignment fix for 32-bit ARM

`Entry.fromListNode` uses `@fieldParentPtr` to recover the parent
struct from a doubly-linked-list node embedded inside it. On 32-bit
ARM, `Entry` (alignment 8 — it has a `u64` field) is wider-aligned
than `std.DoublyLinkedList.Node` (alignment 4 — two pointer fields,
4 bytes each on a 32-bit target). Without an explicit `@alignCast`
the compiler refuses the conversion, since the Node pointer's
declared alignment is narrower than the parent's required alignment.

The Node we receive is always embedded inside a heap-allocated
`Entry`, so the underlying memory is in fact 8-aligned. `@alignCast`
asserts that explicitly. No-op on 64-bit targets.

### CI release pipeline

`.github/workflows/release.yml` runs on tag push (`v*.*.*`) and on
manual dispatch. Three jobs:

1. **test** — `zig build test` on ubuntu-latest, gates the release.
2. **build** — six-target matrix, each producing a single binary
   uploaded as a workflow artifact.
3. **release** — collects the six artifacts, writes `SHA256SUMS`,
   creates the GitHub Release with `docs/release-notes/<version>.md`
   as the body.

Pinned: `actions/checkout@v5`, `mlugg/setup-zig@v2` (zig 0.15.2),
`actions/upload-artifact@v4`, `actions/download-artifact@v4`,
`softprops/action-gh-release@v2`.

## Tests

No new tests — the existing 113/113 suite still passes natively, and
all six targets now compile cleanly under
`zig build -Dtarget=... -Doptimize=ReleaseSafe`.

## Backwards compatibility

Fully backwards compatible. Every CLI invocation that worked in
0.5.1 still works in 0.5.2 unchanged. The on-wire protocol is
identical — peers can't tell which version of zigbee they're talking
to. Existing `~/.zigbee/identity.key` and chequebook state files are
unchanged.

## Not yet — deferred to follow-ups

- **Windows.** Not on the IoT roadmap. Adding it later costs
  one matrix entry; nothing else needs to change.
- **Docker images.** Single-binary `scp` deployment is the IoT
  story. A Docker image is a follow-up if/when there's demand.
- **Stripped / `ReleaseSmall` builds.** ReleaseSafe (~6 MB on
  Linux) is the right default for a v1; `ReleaseSmall` (~1.4 MB)
  pairs with the embedded work in 0.7.
