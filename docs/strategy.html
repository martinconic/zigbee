zigbee · strategy dossier

What zigbee should be — and how it gets there

A consolidated record of the strategy conversation that followed the 0.4 release and the binary-size discussion: research into in-browser Bee clients, an inventory of what blocks zigbee from embedded / wasm targets, the tradeoffs between a full Go-bee rewrite and a small-footprint focus, and the four-milestone roadmap we agreed on.

Captured: 2026-04-28 Repo: github.com/martinconic/zigbee Latest release: v0.4.0 Tests: 62 / 62

Section 1

Binary footprint — same source, four output sizes

The default zig build produces a 23 MB Debug binary. That's full DWARF debug info, no inlining, all safety checks, no dead-code elimination — the right artifact for development but absurd for distribution.

Switching to one of Zig's release modes via -Doptimize=… changes the picture entirely:

Mode Binary What you get
Debug default ~23 MB Full debug symbols, all safety checks, no optimisation. Compiles fast, runs slowly. Dev only.
ReleaseSafe recommended ~6 MB -O3 with Zig's safety checks on. Hostile inputs crash loudly instead of silently corrupting. Right default for a network daemon.
ReleaseFast ~5.5 MB -O3, safety checks off. Slightly faster; appropriate for benchmarks and trusted-data paths.
ReleaseSmall ~1.4 MB -Os, aggressive dead-code elimination. The choice for embedded targets, wasm, low-bandwidth distribution.

All four pass the full test suite (zig build test -Doptimize=…). Release-mode binaries are already stripped; running strip on them saves nothing further.

Key insight

The 16× shrink from Debug to ReleaseSmall isn't gimmicky — it's the difference between "Linux-server only" and "fits on an ESP32 with room to spare". ReleaseSmall is what makes the embedded and in-browser conversations even worth having.

Section 2

Three reference points: how others have tried this

Before deciding what zigbee should target, we surveyed three existing attempts at non-server Bee clients. Each one told us something different about what works, what doesn't, and why.

2.1 vertex (Rust) — abandoned wasm scaffolding

Path: /home/calin/work/swarm/bee-clients/vertex. Vertex is a multi-mode Rust Bee client (bootnode / client / storer-not-yet) with 54 organised crates and Tokio + libp2p underneath. It actually pushes and pulls; client mode is a real read-write node.

What's relevant for us: vertex has two abandoned wasm crates (bin/wasm-playground/, bin/swarm-wasm-lib/) last touched mid-2023. Both are thin wasm-bindgen wrappers around BMT primitives only — no networking, no node, no protocol. The full client uses std::net scattered throughout without conditional compilation, so it cannot compile to wasm without major work.

Lesson Wrapping primitives (BMT, chunk hashes) in wasm-bindgen is the easy part. Compiling a full networking stack to wasm is where projects stall. Vertex is a useful negative example.

2.2 weeb-3 (Rust) — the working in-browser Bee client

Path: /home/calin/work/swarm/bee-clients/weeb-3. This one works. ~10 K lines of Rust compiled via wasm-pack, deployed at lat-murmeldjur.github.io/weeb_3, npm package @lat-murmeldjur/weeb_3 published. Uploads, downloads, integrates with MetaMask for stamp purchases. Live.

Their architecture is the canonical "how to do this":

Layerweeb-3 choiceWhy it works
Crypto alloy + k256 (pure Rust secp256k1) Sidesteps the libsecp256k1-to-wasm pain entirely. No FFI, no Emscripten.
Transport (browser) libp2p with websocket-websys Browser can't open TCP at all. Bee bootnodes already advertise /ws/ multiaddrs.
Concurrency Single-threaded async via wasm-bindgen-futures Browser wasm has no threads (without SharedArrayBuffer + COOP/COEP).
Storage indexed_db_futures Browser has no filesystem. IndexedDB is the persistent option.
Web integration Service Worker proxies swarm:// URL fetches Lets static HTML in the browser pull assets through the wasm Bee.
Stamp purchase Delegated to MetaMask via window.ethereum The wallet handles keys, gas, signing, broadcasting. weeb-3 just describes the tx.
Code split [target.'cfg(target_arch = "wasm32")'.dependencies] Two completely separate dep sets per target. ~99 % of the protocol logic is shared.

Lesson weeb-3 is what an in-browser zigbee would conceptually look like — protocol logic untouched, transport + concurrency + crypto swapped out at the target boundary. The fact that it shipped is the strongest signal we have that the wasm path is viable.

2.3 bee PR #5321 (Go) — full attempt, stalled

github.com/ethersphere/bee/pull/5321 is a draft PR ([DO NOT MERGE], last updated Jan 2026, no formal reviews). 28 commits, +8 514 / −2 528 lines across 41 files. Tries to compile the official Go bee codebase to wasm:

  • Replaces TCP/QUIC with WebSocket via a forked libp2p (github.com/v1rtl/go-libp2p-wasmws).
  • Replaces DNS with DoH for browser compatibility.
  • Splits files by build tag: libp2p_js.go / libp2p_native.go, router_js.go, fs_js.go, node_js.go.
  • Disables WebRTC, AutoNAT, TCP, native filesystem in wasm builds.

It got past compilation and stalled on filesystem abstraction + version-compatibility bugs. Still draft. Never merged.

Lesson Even with Go's ecosystem and the official Bee codebase, retrofitting wasm onto a 250 K-line native codebase is hard enough that it stalls in draft. Weeb-3 succeeded where this didn't because weeb-3 was designed for browser from the start (clean deps, pure-Rust crypto, single-threaded async from day one). zigbee today is in bee-PR-5321's situation, not weeb-3's.

What we take from these three

  • Browser-friendly Bee clients are possible — weeb-3 proves it.
  • The right architecture is platform-conditional dependencies from day one — not retrofit. Pure-language crypto, WebSocket transport, single-threaded async, JS FFI for browser-specific bits.
  • Stamp purchase is delegated to the wallet in browser. We do not need to ship an Ethereum RPC client to push from a browser.
  • Service Workers are how you make Swarm-hosted websites work end-to-end (a static HTML page rendering assets fetched from Swarm references).

For the rest of this dossier we treat the canonical Go bee as the standard — vertex, weeb-3, and bee-PR-5321 are reference points only.

Section 3

What in zigbee blocks each target

Inventory of every place zigbee currently reaches outside pure-Zig code. Numbers are call-site counts across the 27 modules.

SurfaceSitesWhereStory per target
std.Thread (spawn / Mutex / Condition) 21 p2p.zig, connection.zig, yamux.zig (yamux reader, hive dialer, watchdog) browser-critical Browser can't spawn threads without SharedArrayBuffer + COOP/COEP; wasm32-wasi has no threading either. Means full async-state-machine rewrite of the I/O paths for any wasm target.
Vendored libsecp256k1 (C) 54 + 6 in build.zig identity.zig, bzz_address.zig, build linkage non-native critical Pre-built .a is x86-64. To target ARM/MCU/wasm we need to either (a) compile to wasm via Emscripten, (b) call out to JS like noble-secp256k1, or (c) cross-compile to ARM/Cortex-M. Pure-Zig secp256k1 doesn't exist at production quality.
std.net (TCP) 7 connection.zig, noise.zig, dnsaddr.zig replaceable Browser has no TCP; must use WebSocket. MCU needs lwIP. ARM Linux: works as-is.
std.posix (UDP for DNS) 6 dnsaddr.zig replaceable Browser: substitute DoH (fetch()). MCU: hardcode nameservers.
std.fs 3 dnsaddr.zig (/etc/resolv.conf), p2p.zig (write chunk), root.zig (stdout) replaceable Browser: IndexedDB. MCU: SPIFFS / LittleFS / hardcode.
std.crypto 16 Noise, BMT, identity, peer-id, ping fine everywhere All pure Zig (X25519, ChaCha20-Poly1305, SHA256, Keccak, ECDSA-P256, HKDF). No external dependencies.
std.heap.GeneralPurposeAllocator 1 main.zig fine wasm Pure Zig. For MCU swap to a fixed-arena allocator.
std.process.argsAlloc 2 main.zig cli-only Browser has no argv; affects entry point only, not the library. Use URL params or JS API.
Conditional compilation (@import("builtin")) 0 — add when needed No target gates today. Adding them is mechanical but pervasive once we start.

Difficulty by target

TargetVerdictEffortCritical blocker
Embedded Linux ARM (Pi, BeagleBone) trivial 1–2 days Just cross-compile libsecp256k1 for ARM.
Bare-metal MCU (ESP32, RP2040, STM32) doable 2–4 weeks RTOS + lwIP integration; cross-compile libsecp256k1 to bare-metal.
wasm32-wasi (Node.js, Wasmtime) medium 1–2 weeks Threads → async refactor; libsecp256k1 → JS FFI or Emscripten.

Net read

About 5 000 of zigbee's 7 900 lines are pure protocol logic (Noise, Yamux frames, multistream-select, bee handshake, hive, retrieval, joiner, mantaray walker) that compiles to wasm32 unchanged. The remaining ~3 000 lines (transport + connection lifecycle + threading + DNS + secp256k1 calls + main.zig) is what gets a target-specific rewrite.

Section 4

Three strategic options — and what Zig actually buys you

What Zig wins over Go (be honest)

  • Memory: Go bee runs at ~500 MB – 2 GB RSS. Zig equivalent ~80–300 MB. ~3–7× win.
  • Binary size: Go bee ~50–80 MB. Zig 1.4–6 MB. ~10–50× win.
  • Latency tail: No GC pauses. Matters for storers under SLO.
  • Deployment surfaces Go can't reach: bare-metal MCU, freestanding wasm.
  • Predictable allocations: every alloc is explicit. Easier security audit.

What Zig doesn't win meaningfully

  • Throughput. Both within ~1.2× of each other for I/O-bound code. Bee is network-bound. ~1.1× moved chunks/sec.
  • Feature velocity. Go has the mature ecosystem. A 2-person Zig team will not out-ship the Go bee team.
  • Adoption / trust. Bee operators are conservative; they won't switch unless we offer something Go bee can't.

Option A — Full Go-bee parity rewrite

Pitch: "The official Go client, but with 5× less RAM and 10× smaller binary."

Go bee is ~250–300 K Go lines, 7+ years of work. We're at ~7 900 lines, ~15–20 % of the surface area (boundary protocols + read + manifest walking + bee-compatible read-only API). The remaining 80 % is storage, accounting, chain integration, redistribution.

Honest case: Years of work, zero novel capability, perpetual catch-up against an actively-developed Go upstream. The Bee team won't help if we have a bug. Operators have no reason to switch unless they're idealogically motivated.

Verdict Bad ROI as primary scope. Don't aim for parity rewrite as the headline goal. But — gradual progress toward parity is an automatic side effect of doing the other options well.

Option B — Embedded focus (MCU + ARM Linux)

Pitch: "The first Bee client that runs on an ESP32 / Pi Zero. Decentralized storage on every IoT device."

A 1.4 MB binary fits on an ESP32-S3 with room for application code. RAM at ReleaseSmall is 5–20 MB which is borderline ESP32 (with PSRAM) but comfortable on Pi Zero W.

Concrete use cases:

  • IoT sensor uploads readings to Swarm (no central cloud).
  • Drones / robots fetch firmware updates from Swarm references.
  • Industrial / medical edge gateways needing verifiable storage.
  • Smart-home devices presenting Swarm content (digital photo frame, etc.).
Verdict A unique offering — Go bee genuinely can't do this. Plays to Zig's strengths exactly. Defensible niche, ship-able scope.

Option C — In-browser focus (wasm)

Pitch: "Swarm in any tab. No backend. No gateway."

Massive potential reach (every browser). Aligns with Swarm's decentralized-web vision. Service Workers can intercept swarm:// URLs across the web.

Honest case: weeb-3 already does this. To beat it, zigbee.wasm would need to be:

  • Smaller — likely yes (Zig has no runtime; weeb-3's bundle is ~3 MB; we could plausibly hit 1–1.5 MB).
  • Faster — likely a wash (WebSocket I/O is the bottleneck, not CPU).
  • Better DX — subjective.
Verdict Worth doing eventually, but you're entering an existing market. Marginal-improvement play, not category-defining the way embedded is.

Section 5

The four-milestone roadmap

Don't pick one option — sequence them. The framing: zigbee is the small-footprint Bee client family. Read-only today, write-capable tomorrow, deployable on any target Go can't fit on.

MilestoneTagFocusEffort
0.5 retrieval-maturity SWAP cheques + SOC validation + encrypted refs + persistent identity + dead-conn pruning + basic local store ~2–3 months
0.6 push postage stamp parser + issuer + /swarm/pushsync/1.3.1 + upload API; accepts a batch credential as input ~3–4 months
0.7 embedded ARM Linux first-class (release matrix); MCU spike (ESP32-S3) optional ~1–2 months
0.8+ browser wasm32-freestanding via WebSocket; libsecp256k1 strategy decided; wallet-delegated stamp purchase revisit after 0.7

0.5 — Retrieval maturity

The headline piece is SWAP: today bee disconnects us with apply debit: disconnect threshold exceeded after ~25–30 unpaid chunks per peer. The protocol is /swarm/swap/1.0.0/swap; we issue cheques to bee peers when our debt crosses a threshold. We can defer cashing cheques on-chain entirely for this milestone — we only need to issue.

  • Cheque issuance + verification (existing secp256k1 + EIP-191).
  • SOC validation (close the chunk-validation gap; needed for feeds later).
  • Encrypted-chunk references (refLength = 64).
  • Persistent libp2p identity on disk.
  • Dead-connection pruning.
  • Basic flat-file chunk store (foundation for 0.6).

0.6 — Push capability

The clean separation: 0.6 ships protocol-only push, no chain integration. Users provide a postage batch credential (id + signing key + metadata) acquired by some other tool — bee, a stamp service, or a wallet. zigbee uses the credential to issue stamps for chunks it pushes.

  • Postage stamp parser + verifier (we already have secp256k1).
  • Postage stamp issuer — sign chunk addresses against a loaded batch credential.
  • Bucket-index tracking (each batch has 2bucket_depth slots; we have to avoid reuse).
  • /swarm/pushsync/1.3.1/pushsync initiator.
  • HTTP API: POST /bytes, POST /bzz (with manifest building).

What's deferred to 0.6.1 / 1.0:

  • Buying batches via Eth RPC (full chain integration).
  • Cashing cheques on-chain.

0.7 — Embedded

  • Cross-compile libsecp256k1 for arm-linux-gnueabihf and aarch64-linux-gnu.
  • Validate end-to-end on Raspberry Pi Zero W (most demanding target).
  • CI release matrix: x86_64 / armv7 / arm64 Linux static binaries.
  • Optional MCU spike: ESP32-S3 with FreeRTOS + lwIP; cross-compile libsecp256k1 to Xtensa. Real work; do only if there's a use case.

0.8+ — In-browser (revisit)

After 0.7 ships, decide based on:

  • Did embedded validate real demand for non-server zigbee?
  • Has weeb-3 cornered the in-browser market by then, or is there room?
  • Have we decided libsecp256k1-vs-pure-Zig (which embedded MCU also forces us to confront)?

If we go ahead, the work is: transport abstraction (TcpTransport vs WebSocketTransport), single-threaded async event loop, JS FFI for browser APIs, Service Worker bridge.

Critical observation

The 5 000 lines of pure protocol logic (Noise, Yamux, multistream-select, bee handshake, hive, retrieval, joiner, mantaray) are shared across every milestone and every target. Only the ~3 000 lines of transport / threading / fs / crypto adapters change per target. Designing for that split is the actual architectural work; the protocol code is already done.

Section 6

Chain integration as an outer ring, not a core

Three places where Bee touches Ethereum:

  1. Buy postage stamps — call PostageStampContract.createBatch(...), costs gas + BZZ.
  2. Cash SWAP cheques — call Chequebook.cashCheque(...), costs gas.
  3. Read-only RPC calls — query stamp validity, contract state. No gas.

The right architecture is to treat chain integration as an outer ring around zigbee, not part of zigbee itself. The core stays small and target-portable; chain code is an optional adapter layered per target.

Sub-taskServerBrowserEmbedded MCU
Manage a private key We sign keys, store, protect free — wallet has the keys Real key-management problem
Hold ETH (gas) and BZZ Operator's responsibility free — user's wallet Prepaid balance or external funder
Build & sign tx Eth RPC client + secp256k1 + RLP encoding (~5 K lines) free — wallet builds & signs Same as server-side, on tiny hardware
Broadcast tx Connect to RPC endpoint (Infura, etc.) free — wallet broadcasts RPC over HTTPS — feasible but adds TLS
Encode contract calldata Need an ABI encoder Tens of lines — describe the call to the wallet Same as server-side

Counterintuitive insight The browser is the easiest place to do stamp purchase, not the hardest. MetaMask handles keys, gas, signing, broadcasting, and the UX. We just describe what tx to construct. weeb-3 proves this pattern works.

Per-target story

Browser (0.8) — wallet bridge
  • Read state: window.ethereum.request({method: 'eth_call', params: [...]}).
  • Buy stamp: encode createBatch(...) calldata, call eth_sendTransaction, wait for receipt.
  • Cheque cashing: same pattern.
  • ~few hundred lines of JS FFI; key handling, gas estimation, popup UX → wallet's job.
Server (1.0) — own RPC client
  • Full Ethereum RPC client in Zig.
  • Key management on disk (encrypted keystore).
  • Postage / chequebook contract bindings.
  • ~5–10 K lines of Zig. Real chunk of work; this is the "approaches Go-bee parity" milestone.
Embedded — stay external
  • No on-device chain integration.
  • Use pre-flashed batch credentials, or fetch credentials from a backend stamp service.
  • Device signs stamps locally (one secp256k1 sign per chunk — cheap, fast).
  • Never broadcasts a transaction.
Net effect Chain integration never blocks any of 0.5 / 0.6 / 0.7. It becomes a target-specific add-on: browser gets it via wallet (cheap), server gets it via Eth RPC (expensive — defer to 1.0), embedded skips it (correct by use-case design).

Section 7

Concrete example — pushing from an ESP32

The pushsync wire format is, per chunk, Delivery { address, data, stamp }. The stamp is a secp256k1_sign(batch_private_key, …) over the chunk address plus batch metadata. The receiving bee verifies that signature against the batch owner's address recorded on-chain.

Key fact Issuing a stamp is a free off-chain operation: one secp256k1 signature per chunk. No RPC call, no gas, no transaction. The on-chain cost (BZZ + gas) is paid once, at batch-creation time, by whoever funded the batch. The device pushing chunks never touches Ethereum.

So an ESP32 needs three things to push:

  1. Batch ID (32 bytes — public).
  2. Batch's private signing key (32 bytes — must stay secret on the device).
  3. Batch metadata (depth, bucket_depth, validity window).

Three provisioning patterns

A — Pre-flash a long-life batch into NVS at deploy time
At build / deploy time, on a laptop with bee:
   1. Buy a batch with bee:  POST /stamps/<amount>/<depth> → returns batch_id
   2. Export the batch's signing key + metadata to a small blob.
   3. Flash that blob into the ESP32's NVS / SPIFFS partition.

On the device, every push:
   1. Hash data → chunk addresses (BMT, already in zigbee)
   2. For each chunk: zigbee.signStamp(batch, chunk_addr) — local secp256k1
   3. pushsync.send(closest_peer, Delivery { addr, data, stamp })
   4. Verify Receipt

A depth-22 batch costs ~$5–15 mainnet, signs ~4 M stamps, lasts months. Plenty for a sensor uploading every 10 minutes for a year.

B — Backend stamp service

For higher-throughput devices (cameras, drones, multi-tenant fleets): 

ESP32 boots → connects to backend HTTPS → POST /provision { device_id }
   ← { batch_id, key, depth, valid_until } (TLS-encrypted)
ESP32 caches credential in flash, uses until expiry, refreshes.

The backend holds the batches, possibly buys new ones as they expire — that's where chain integration lives, on the backend, not the device. Standard IoT-to-cloud auth patterns apply (mTLS, device certificates).

C — Stamp-on-demand (per-chunk RPC)

Useful when you don't want the batch key on the device at all (defence-in-depth): 

For each chunk: device → backend POST /sign { chunk_address }
                       ← { stamp_signature, batch_id, bucket_index }
device.pushsync.send(peer, Delivery{ addr, data, stamp })

Adds a network round-trip per chunk; OK at low rates, kills throughput for video. Real benefit: device compromise doesn't compromise the batch.

Worked example: temperature sensor uploading every 10 minutes

Batch:        depth=22, bucket-depth=16, TTL ≈ 12 months  (one-time buy: ~$10 mainnet)
Chunks/year:  144 readings/day × 365  ≈  52 600  (each reading = 1 chunk; small)
Stamp use:    ~52 600 over the year
Capacity:     2^22 × 2^16 = 2.7 × 10^11  → vastly oversized, can host thousands of devices
Per-chunk:    1 secp256k1 sign (~5 ms on ESP32) + handshake reuse + 1 pushsync round-trip
Memory:       zigbee ReleaseSmall ~1.4 MB code + ~8 KB per chunk RAM
Network:      idle most of the time; pushes a single chunk every 10 min
On-chain:     never. Funding event happened once on a laptop.

The device pushes happily for a year on one batch. When the batch nears expiry an OTA update or backend round-trip refreshes the credential. The device never touches a chain RPC, never holds ETH, never broadcasts a transaction.

Small reframing of the 0.6 design

The ESP32 case clarifies that 0.6 needs to be slightly more than "accept stamps from outside" — it needs a stamp issuer, not just a parser/verifier:

ComponentWhat it doesNew code?
Postage batch loader Read {batch_id, key, depth, valid_until} from a blob/file/env var ~100 lines
Stamp issuer signStamp(batch, chunk_addr, bucket_idx) → 65-byte sig Trivial — uses existing identity.signEthereum
Stamp verifier Recover address from sig, check against batch owner Trivial — uses existing identity.recoverEthereum
Bucket-index tracking Avoid reuse within a bucket ~150 lines + persistence
/swarm/pushsync/1.3.1/pushsync Wire protocol: Delivery → Receipt ~300 lines
HTTP API POST /bytes, POST /bzz (with manifest building) Existing pattern (~400 lines)

Section 8

Open questions before each milestone

For 0.5 (retrieval maturity) — small scope decisions

  • Do we cash any received SWAP cheques on-chain in 0.5, or strictly defer to 1.0? (Recommendation: defer.)
  • How aggressive is the local store? Just an LRU cache, or a real persistent store with sweep / eviction? (Recommendation: minimal LRU; durability is 0.6/0.7 work.)
  • Are we OK with bucket-index reuse in the issuer until 0.6 needs it strict?

For 0.6 (push) — credential UX decisions

  • How is the batch credential supplied — JSON file, environment variable, HTTP endpoint, all of the above?
  • Do we generate the batch's signing key client-side (zigbee never sees the funded key), or accept it directly? (The former is more secure; the latter is simpler.)
  • Pushsync receipt verification: do we cryptographically verify the receipt's signature, or trust the protocol?

For 0.7 (embedded) — target choices

  • Which MCU first if we do the spike? ESP32-S3 (PSRAM, Wi-Fi/BT built in) is the easiest answer, but RP2040 is more "open hardware".
  • FreeRTOS or Zephyr for the RTOS layer? (FreeRTOS is what ESP-IDF wants; Zephyr is more portable.)
  • Is the use-case real enough to justify 3–4 weeks of MCU work, or do we ship ARM Linux only and revisit MCU when there's a customer?

For 0.8 (browser) — strategic decisions

  • Pure-Zig secp256k1 (a multi-week project of its own), Emscripten-compiled libsecp256k1, or JS FFI to noble-secp256k1? (JS FFI is fastest; pure-Zig is best long-term.)
  • Service Worker integration in v1, or strictly the wasm + SDK in v1 with Service Worker in v2?
  • Do we ship our own demo UI (like weeb-3 does) or focus on the SDK and let dApps build their own?

The bigger strategic forks (revisit periodically)

  • Who's the ideal user in 2 years? An IoT developer, a dApp author, a Bee operator? The answer changes the priority order.
  • Funding / team scale? Solo developer can ship the read-only matrix (Linux/ARM/MCU/browser) in 12–18 months. A team could attempt full Go-bee parity.
  • What does "Swarm adoption" look like to you? Every dApp consuming Swarm in the browser → prioritise wasm. Every IoT device storing telemetry → prioritise embedded. More independent operators → prioritise full-node parity.
