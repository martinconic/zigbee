zigbee · IoT roadmap

zigbee for embedded — what we build, why, and in what order

The 0.5 → 0.6 → 0.7 sequence locked in by the strategy dossier happens to be exactly the path to a working IoT story. This page re-frames each work unit through "what does it buy a deployed embedded device", flags three things the original plan under-specified for IoT, and adds the cross-cutting items and demo tracks that prove the IoT use case end-to-end.

Captured: 2026-04-28 Repo: github.com/martinconic/zigbee Latest release: v0.4.0 Headline focus: IoT / embedded

Section 1

IoT is the headline focus — what changes, what doesn't

Going forward, the framing of zigbee is:

zigbee is the small-footprint Bee client family for devices that can't run Go bee. Servers and browsers are valid targets but not the headline; the IoT / embedded case drives priorities.

What that doesn't change:

  • The four-milestone sequence (0.5 retrieval-maturity → 0.6 push → 0.7 embedded → 0.8+ browser). Each milestone is a hard prerequisite for IoT — you can't build an IoT story on a client that can't get past the 25-chunk-per-peer accounting wall, or that can't push.
  • The "no chain integration in zigbee proper" architecture. The browser-via-MetaMask pattern stays valid; the server-via-own-RPC pattern stays valid; neither is on the IoT critical path.
  • The "accept-credential-from-outside" model for postage stamps. Patterns A (pre-flash), B (backend service), C (per-chunk RPC) from strategy.html §7 cover the realistic IoT deployment models.

What it does change:

  • 0.7b ESP32-S3 spike is no longer "gated on use case" — it's planned. If IoT is the destination, MCU validation is a milestone, not an aspiration.
  • Five cross-cutting concerns become first-class — resource bounds, persistent state durability, continuous cross-compilation in CI, production logging, static-link path. None are tasks today; all are easy to retrofit by accident if we don't track them. See §4.
  • Three demo / proof-of-concept tracks are added, each one a concrete IoT scenario that proves the plumbing works end-to-end. See §5.
  • Within each milestone, IoT-essential items are prioritised over IoT-medium ones. E.g. persistent identity (essential) lands first; SOC validation (medium) lands later in the same patch.

Task table — annotated for IoT relevance

Live from PLAN.md §9, with a priority column for what each item buys you specifically on a deployed embedded device.

Priority legend: essential = blocks IoT outright. important = real-world reliability / cost. medium = nice to have; defer if pressed. skip = irrelevant for IoT.

# Work unit Effort Priority Why for IoT
0.4.1 patch — small wins
0.4.1a Persistent libp2p identity → ~/.zigbee/identity.key 1–2 d essential Reboots are routine for IoT. A device that loses its identity on every reboot loses bee accounting state every time.
0.4.1b Dead-connection pruning 2–3 d important A daemon meant to run for months on an ESP32 cannot leak memory on every connection drop.
0.4.1c SOC validation 3–5 d medium Defence-in-depth. Required prep for reading feeds (a future IoT capability — devices subscribing to feed updates).
0.5.0 — retrieval maturity (~10 work-weeks FTE)
0.5a Local flat-file chunk store (basic LRU) ~1 wk essential IoT devices have intermittent connectivity. A local cache means firmware updates / config can be re-read offline. Also lets one device act as a tiny Swarm cache for nearby devices on the same LAN.
0.5b Encrypted-chunk references (refLength = 64) ~2 wk medium Useful when telemetry is sensitive (medical, industrial). Not required for unencrypted public data.
0.5c SWAP cheques (issue-only) ~4–5 wk essential Without SWAP a device can pull at most ~25–30 chunks per peer per session — about 100 KB. Useless for firmware (~MB) or sustained sensor sync. The single biggest IoT unblocker.
0.5.0 Cut release — — —
0.6.0 — push (~12 weeks FTE; the IoT-headline milestone)
0.6 design Accept-credential-from-outside model — protocol-only push, no chain code in zigbee — essential This is the IoT story. Device gets a batch credential, signs stamps locally, never touches Ethereum. See strategy.html §7.
0.6a-i Postage batch loader ~100 LOC essential Reads the credential the device receives via patterns A/B/C.
0.6a-ii Stamp issuer ~50 LOC essential Sign stamps locally per chunk. Reuses our existing secp256k1.
0.6a-iii Stamp verifier ~50 LOC medium Verify receipts on chunks we push (proves the storer received them).
0.6a-iv Bucket-index tracking + persistence ~150 LOC + flash essential Bee rejects stamps with reused bucket indices. On a device that reboots, this state must survive across power cycles or every reboot risks rejected pushes.
0.6b /swarm/pushsync/1.3.1 initiator + receipt verification ~300 LOC essential The actual push protocol. Without this there's no upload at all.
0.6c-i HTTP POST /bytes upload ~250 LOC important Useful for IoT clients that produce raw blobs. May not be needed if device uses zigbee in-process (as a library) instead of via HTTP.
0.6c-ii HTTP POST /bzz + manifest building ~150 LOC medium Manifests matter when references need filenames / paths. IoT often doesn't — sensor reading at <hash> is enough.
0.6.0 Cut release — — —
0.7.0 — embedded (the runtime enablement)
0.7a ARM Linux release matrix ~1 wk essential The easy IoT target — Pi Zero, BeagleBone, OpenWRT routers, edge gateways. Cross-compile + CI artifacts.
0.7b ESP32-S3 spike (planned, not gated) ~4 wk essential The hard IoT target — actual microcontrollers. Re-classed from "gated on use case" to "planned" now that IoT is the headline focus.
0.7.0 Cut release — — —

Section 3

Three things the original plan under-specified for IoT — now closed

Gap 1 — 0.7b ESP32-S3 was "gated"; now planned

PLAN.md previously said "ESP32-S3 spike — gated on use case". If IoT is the stated focus, that is the use case. The spike is now planned and on the milestone path, with effort estimated at ~4 weeks (FreeRTOS + lwIP + Xtensa-cross-compiled libsecp256k1 + thread→task replacement + GPA→pool allocator).

Gap 2 — Cross-cutting IoT items were missing

Five concerns that fall across milestones rather than inside one. None were tracked anywhere; all are easy to retrofit by accident if we don't surface them. See §4.

Gap 3 — Demos / proof-of-concept tracks weren't called out

The IoT story is sold by working examples, not by a feature list. Three concrete scenarios — temp sensor, Pi Zero gateway, firmware update from Swarm — are now tracked separately from the milestone work units. See §5.

Section 4

Cross-cutting IoT items — added to the plan

These don't belong inside any single milestone but they have to land on a schedule, otherwise we get a daemon that nominally "works on a Pi" but in practice can't be deployed at scale.

#ItemLands inWhat it specifically prevents
X1 Resource bounds — make every queue / buffer / cap configurable. CLI flags: --max-peers, --max-in-flight-chunks, --store-max-bytes, --http-max-body. Document recommended settings per target (server / Pi / ESP32). start in 0.5; finalise in 0.7a An ESP32 with 8 MB PSRAM cannot run with the same buffer defaults as a desktop. A lazy "let it grow" answer is what kills IoT deployments.
X2 Persistent state survives crash / power loss — atomic write + journaled recovery for: identity, local store, bucket-index counters, cheque ledger. Tested by killing the process during writes; on recovery, partial state is rolled back, not silently corrupted. folded into 0.4.1a + 0.5a + 0.6a-iv An IoT device that reuses a bucket index after an unclean reboot starts having its pushes silently rejected. An identity file partially overwritten = lost peer.
X3 Continuous cross-compilation in CI from the start of 0.5. GitHub Actions matrix: x86_64-linux-musl, aarch64-linux-musl, arm-linux-musleabihf. Every commit. vendor/secp256k1 rebuilt for each target. introduce in 0.5; formalise in 0.7a Don't wait for 0.7 to discover something doesn't cross-compile. Catches Linux-isms (e.g. accidental glibc-only syscalls, vendored library assumptions) before they pile up across multiple modules.
X4 Production logging mode — log levels (silent / error / info / debug) + structured (JSON one-line) or human output. CLI flag --log-level. Daemon-mode default = error. early 0.6 A daemon that logs every retrieval attempt at debug level fills an MCU's log buffer in minutes. Production deployments need silence by default.
X5 Static-link path — produce binaries that link statically against musl + vendored libsecp256k1. Single-file scp zigbee <device>:/usr/local/bin/ deploys. 0.7a Alpine, OpenWRT, busybox-based ARM Linux distros either don't have glibc or have an old version. Static binaries deploy on every Linux-like target without dependency drama.
Common thread

All five are about operability: making zigbee something you can actually ship to a device and forget about for a year, not something that runs in your dev shell. None are protocol work; all of them prevent specific real-world IoT failure modes.

Section 5

Proof-of-concept demos — three end-to-end IoT scenarios

Each one a self-contained example sitting under examples/ in the repo. Goal: prove the IoT use case works end-to-end, not just that the protocol implementation passes tests.

Demo 1 — ESP32-S3 temperature sensor → Swarm

Lands after 0.6 + 0.7b. Path: examples/esp32-tempsensor/.

  • BME280 sensor wired to ESP32-S3 dev board.
  • Reading every 10 minutes; pushed to Swarm via zigbee as a CAC chunk.
  • Pre-flashed batch credential (provisioning pattern A from strategy.html §7).
  • Resulting reference printed via UART — concrete proof that an ESP32 can publish to Swarm autonomously.
  • Reference scenario for blog posts / demos / pitches.

Demo 2 — Raspberry Pi Zero retrieval gateway

Lands after 0.5 + 0.7a. Path: examples/pi-zero-gateway/.

  • Pi Zero W on local Wi-Fi, running zigbee daemon.
  • Bee-compatible HTTP API exposed on 0.0.0.0:9090 (bound to the LAN).
  • Other devices on the LAN — phones, laptops, smart-home hubs — hit the Pi for Swarm content.
  • Local chunk store (0.5a) caches frequently-fetched chunks.
  • Demonstrates "one Pi serving Swarm to a household / small business". Useful as a privacy-preserving pinned-content gateway.

Demo 3 — Firmware update from a Swarm reference

Lands after 0.5 + 0.7a (ARM Linux device). Path: examples/firmware-update/.

  • Embedded device boots, reads /etc/zigbee/firmware.ref (a 64-char hex Swarm reference).
  • Retrieves the firmware blob via zigbee from any reachable bee peer.
  • Verifies a signature against a baked-in public key.
  • Triggers a reflash via the device's own update mechanism.
  • Shows the "OTA from a decentralised store" story — central-server-free firmware delivery.

Why these three They cover the three honest IoT shapes: upload-only (sensor → Swarm), download-only (firmware ← Swarm), and two-way / share-with-LAN (Pi gateway). Together they prove zigbee is useful at the device edge, not just on a developer's desk.

Section 6

What an IoT deployment actually looks like

Three deployment shapes, each with a clear separation between "where the BZZ is paid" and "what the device does". The device never spends gas, never holds ETH, never broadcasts a tx.

Shape 1 — Solo maker / hobbyist

┌────────────────┐    one-time           ┌─────────┐
│ developer's    │──── buy batch ─────▶  │ bee on  │
│ laptop (bee)   │                        │ mainnet │
└─────────┬──────┘                        └─────────┘
          │ export {batch_id, key, depth, …}
          │ flash via UART / OTA
          ▼
┌────────────────┐
│  ESP32-S3      │  zigbee.signStamp(batch, addr)
│  (zigbee +     │  pushsync to nearest bee peer
│   sensor)      │  ─────────────────────────────▶ Swarm
└────────────────┘
   forever, no chain access

Shape 2 — Small fleet (10-1000 devices)

┌──────────────────┐    on-chain ops     ┌─────────┐
│ stamp-service    │──── buy batch ────▶ │ bee     │
│ backend (HTTPS)  │                      └─────────┘
└────────┬─────────┘
         │ device → POST /provision
         │ ← {batch_id, key, depth, valid_until}
         ▼
┌──────────────────┐         ┌──────────────────┐
│  device #1       │         │  device #N       │
│  (zigbee)        │   …     │  (zigbee)        │
└────────┬─────────┘         └────────┬─────────┘
         └───── Swarm via pushsync ────┘

Shape 3 — High-security / per-chunk RPC signing

┌──────────────────┐    holds batch key  ┌─────────┐
│ signing service  │      (HSM)          │ bee     │
└────────┬─────────┘                      └─────────┘
         ▲ device → POST /sign {chunk_addr}
         │ ← {stamp signature}
         │
┌────────┴─────────┐
│  device          │  pushsync to bee peer
│  (zigbee, no key)│  ────────────────────▶ Swarm
└──────────────────┘

All three are supported by the same zigbee binary; the differences live in where the credential comes from and how often the device fetches it — not in zigbee's code path. The device itself is read-mostly + sign-only.

Concrete reference numbers Depth-22 batch on mainnet ≈ $5–15. Capacity: 222 × 216 = 2.7 × 1011 stamp slots — vastly oversized for a single device. A temperature sensor at one reading every 10 min consumes about 52 600 stamps a year. Sub-cent per device per year. Sustainable for fleets of thousands on a single batch.

Section 7

How this fits the other docs

Five docs in the repo carry the IoT story now, layered from high-level to operational:

  • strategy.html — the original strategic dossier. Research findings (vertex, weeb-3, bee PR #5321), the three options analysis, the four-milestone roadmap, the chain-integration model. Written once; doesn't change with the IoT re-framing — IoT is one of the three options that happened to win.
  • iot-roadmap.html (this page) — the IoT-specific re-framing. Read alongside strategy.html.
  • PLAN.md §9 — the live tracking table. Updated when phases ship. The table in §2 of this page is sourced from there.
  • STATUS.md — the operational snapshot, with the resumption checkpoint at the top. Refreshed every time state meaningfully changes.
  • README.md, USAGE.md, ARCHITECTURE.md — user-facing docs. They evolve as features land; the IoT framing will surface in them per release.

When we cut 0.5, 0.6, 0.7 — the IoT angle gets release notes of its own (RELEASE_NOTES_0.5.md, etc.) so the "what does this mean for embedded" framing stays explicit.
